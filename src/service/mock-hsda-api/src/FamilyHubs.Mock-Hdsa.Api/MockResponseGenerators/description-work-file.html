Introduction

A mock server implementing v3 of the Human Services Data Specification (HSDS), part of the Open Referral Initiative.

How to use

Mock response keys

Mock responses are fetched from the database. Each mock response is keyed according to the:

  • operation name
  • scenario name
  • path parameters
  • query parameters

Operation name

The operation names supported by the HSDS are:

  • getAPIMetaInformation
  • getFullyNestedServiceById
  • getPaginatedListOfServices
  • getTaxonomyById
  • getPaginatedListOfTaxonomies
  • getPaginatedListOfTaxonomyTerms
  • getTaxonomyTermById
  • getOrganizationById
  • getPaginatedListOfOrganizations
  • getServiceAtLocationWithNestedDataById
  • getPaginatedListOfServiceAtLocation

Scenario name

By default, when an endpoint is called, the server will return the mock response with a null scenario name.

If the request includes a header called X-Mock-Response-Id, the server will return the mock response with the matching scenario name of the given header value.

Path parameters

If the endpoint has path parameters, the server will return the mock response with the matching path parameters. The path parameters should be in the database as a string containing key=value pairs separated by &, ordered alphabetically by key. For example, getFullyNestedServiceById has a path parameter of id, so to key the mock response for the id of ac148810-d857-441c-9679-408f346de14b, the path parameter would be id=ac148810-d857-441c-9679-408f346de14b.

Query parameters

If the endpoint supported query parameters, the server will return the mock response with the matching query parameters. The query parameters should be in the database as a standard query parameter string (without the initial '?'), except the keys should be sorted alphabetically. So basically, the path and query parameters are stored in the database in the same format.

Mock responses

Each mock response consists of the:

  • status code
  • response body

Custom response headers

The CORS response header recommended by the HSDS is added to each response.

Development notes

This server has been implemented as a generic mock HTTP server, with HSDS specific customisations dependency injected in via the HsdaPagingMockResponseGenerator class.

Seeded mock responses

The database is seeded with some example mock responses. Here are some of the example mock responses for the getPaginatedListOfServices endpoint:

Scenario name Description Curl
Pagination Returns a large set of services, useful for testing pagination. The server supports different incoming paging parameters. curl \"https://localhost:7298/services?page=2&per_page=10\" -H \"accept: application/json\" -H \"X-Mock-Response-Id: Pagination\"
Empty Returns an empty set of service.
500 Returns a 500 error response. curl -w \"\nHTTP Status Code: %{http_code}\n\" \"https://localhost:7298/services\" -H \"accept: application/json\" -H \"X-Mock-Response-Id: 500\"
